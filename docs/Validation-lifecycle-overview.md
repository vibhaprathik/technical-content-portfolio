# Validation Lifecycle Overview

## Introduction

A computerised system validation lifecycle is used to ensure that a system is fit for its intended use and operates in a controlled and documented manner.

This lifecycle is especially important where systems support regulated, quality-critical, or business-critical activities.

## Typical stages

Although organisations may use different terminology, a validation lifecycle often includes the following stages:

### 1. Planning
The scope, approach, responsibilities, and deliverables are defined.

Typical documents:
- Validation Plan
- project scope information

### 2. Requirements and risk assessment
User needs are captured and key risks are assessed.

Typical documents:
- User Requirements Specification (URS)
- High-Level Risk Assessment (HLRA)
- Functional Risk Assessment (FRA)

### 3. Design and traceability
Requirements are linked to testing and implementation activities.

Typical documents:
- Requirements Traceability Matrix (RTM)
- configuration or design records

### 4. Testing
The system is tested to confirm that it performs as intended.

Typical documents:
- Installation Qualification (IQ)
- Operational Qualification (OQ)
- Performance Qualification (PQ)
- test scripts and evidence records

### 5. Review and release
Results are reviewed and the system is approved for use where acceptable.

Typical documents:
- summary reports
- deviation records
- approval records

### 6. Ongoing control
After implementation, the system continues to be managed through change control, incident management, and periodic review.

Typical documents:
- change controls
- deviations
- periodic review records

## Why the lifecycle is important

The validation lifecycle provides a structured framework for demonstrating that a system:

- meets defined requirements
- has been assessed for risk
- has been tested appropriately
- is supported by documented evidence
- remains controlled throughout use

## Documentation principles

Strong validation documentation should be:

- clear
- traceable
- consistent
- reviewable
- aligned with procedure and system risk

## Conclusion

Validation is not a single document or a one-time test activity. It is a structured lifecycle supported by linked documents, evidence, and review steps that demonstrate system control and intended use.
